**项目名称：Go-Flash-Job 高性能分布式任务调度引擎**

**项目背景**：
在现代互联网应用中，定时任务（如每日报表生成、用户消息推送、周期性数据清理）的规模和频率日益增长。传统的基于数据库轮询的调度方式存在**性能瓶颈高、资源浪费严重、实时性差**等问题。本项目旨在通过现代化的架构设计，解决以上痛点。

**技术选型**：

- **开发语言**：Go
- **Web 框架**：Gin
- **ORM**：GORM
- **全局调度队列**：Redis (ZSet)
- **任务指令通道**：RabbitMQ
- **日志数据总线**：Kafka
- **数据持久化**：MySQL

1.  **【架构设计】借鉴 GMP 模型，实现无忙等待的调度核心**：
    - **痛点**：摒弃 `for { sleep() }` 轮询数据库或 Redis 的低效模式。
    - **我的方案**：创新性地将 Go GMP 调度思想应用于业务场景。设计了 **Redis ZSet (作为全局待调度任务G队列) + Go 内存最小堆 (作为调度器本地P队列)** 的两级调度架构。
    - **实现细节**：后台协程（P）通过 **Lua 脚本** 以“工作窃取”模式从 Redis 批量预拉取即将到期的任务放入本地堆。主调度协程（M）则利用 `time.NewTimer` 计算与堆顶任务的时间差进行**精准挂起**，实现了CPU的零空转，在无任务时几乎不消耗任何资源。

2.  **【高并发设计】构建基于消息队列的全链路背压系统**：
    - **痛点**：瞬时百万级任务同时触发，下游执行器若无限制地创建 Goroutine 会立刻导致内存溢出 (OOM)。
    - **我的方案**：设计了一套从 MQ 到业务执行的完整流量控制链路。
    - **实现细节**：首先利用 **RabbitMQ 的 QoS (Prefetch Count)** 机制，限制每个执行器节点最多只能接收 N 条未确认的消息。其次，在执行器内部手写了基于 **带缓冲 Channel (作为信号量)** 的轻量级**协程池**。当池满时，新任务提交会自然阻塞，进而暂停从 RabbitMQ 拉取新消息，形成完美的**反向压力传导 (Backpressure)**，保证了服务在任何流量冲击下的绝对稳定性。

3.  **【性能优化】通过批量处理将 IO 开销降至最低**：
    - **痛点**：海量任务的创建和执行日志的写入会产生巨大的网络和磁盘 I/O。
    - **我的方案**：在所有关键 IO 路径上都应用了批量处理思想。
    - **实现细节**：在压测接口中，使用 **Redis Pipeline** 技术将 10 万次 `ZADD` 操作合并为一次网络往返。在日志落库服务中，设计了**内存 RingBuffer + 定时/定量**两种触发条件的**异步批量刷盘 (Batch Insert)** 机制，极大降低了对 MySQL 的写连接压力。

4.  **【高可靠设计】保证核心数据在分布式环境下的不丢失**：
    - **痛点**：分布式系统中，网络抖动或节点宕机可能导致任务或日志丢失。
    - **我的方案**：为不同类型的数据选择了最合适的中间件和策略。
    - **实现细节**：对于**任务指令**，利用 RabbitMQ 的持久化特性和**手动 Ack 机制**，确保任务在被成功执行后才从队列中移除，实现了**“至少一次”**的投递语义。对于**执行日志**，选择吞吐量极高的 Kafka 作为日志总线，利用其**异步生产者**实现日志的快速写入，即使下游的数据库写入服务暂时宕机，日志依然安全地积压在 Kafka 中，不会丢失。

5.  **【高可用设计】引入自动清理机制，保障系统长期稳定**：
    - **痛点**：日志数据无限增长最终会导致磁盘空间耗尽，拖垮整个系统。
    - **我的方案**：在日志清洗服务中内置了一个**守护协程**。
    - **实现细节**：该协程作为一个后台 Cron Job，每日凌晨定时执行 SQL，**物理删除或归档** N 天前的过期日志数据，实现了数据生命周期的自动化管理，保证了数据库的性能和磁盘的可用性。
